# Annotation API v1 (frozen)

> **OLO 1.0 architecture freeze:** [OLO_1_0.md](OLO_1_0.md) — full stack boundaries, catalog resources, runtime vs UI split, and where effort goes next. This document covers the **annotation attribute surface** only.

OLO 1.0 freezes the compile-time metadata surface below. New attributes require a catalog schema version bump and coordinated UI/processor updates.

Full reference: [ANNOTATIONS.md](ANNOTATIONS.md)  
Generated JSON: [CATALOG_SCHEMA.md](../../olo-annotation-processor/docs/CATALOG_SCHEMA.md)

## Frozen surface

| Annotation | Attributes |
|------------|------------|
| **`@OloNode`** | `examples`, `featured`, `deprecated`, `experimental` (+ existing: `type`, `name`, `description`, `category`, `emoji`, `tags`, ports, properties, capabilities) |
| **`@OloTool`** | `examples`, `featured`, `deprecated`, `experimental` |
| **`@OloHook`** | `deprecated`, `experimental` only — no `featured` or `examples` |
| **`@OloPort`** | `description` |
| **`@OloProperty`** | `label`, `placeholder`, `group`, `order`, `secret`, `examples`, `help`, `type` (`OloPropertyType`) |
| **`OloPropertyType`** | `STRING`, `TEXTAREA`, `NUMBER`, `BOOLEAN`, `ENUM`, `JSON` |

## `examples` semantics

On nodes, tools, and properties: **use-cases**, not configuration values.

| `examples` | Not `examples` — use |
|------------|---------------------|
| `Summarize document` | `Summarize the following article…` → `placeholder` |
| `Call REST API` | `https://api.example.com/…` → `placeholder` / `defaultValue` |

See [EDITOR_CONVENTIONS.md](EDITOR_CONVENTIONS.md#examples--use-cases-not-configuration-values).

## Intentionally not in v1

The current model supports palette generation, search, categorization, property forms, grouping, onboarding, recommendations, and in-editor documentation. **Enough for 1.0.**

| Concept | Reason |
|---------|--------|
| `difficulty`, `priority`, `rank` | Subjective; hard to maintain |
| `min`, `max`, `pattern` | Reserved validation — post-v1 |
| `subcategory`, `icon`, `color` | UI polish — not needed yet |
| `OloPropertyGroup` enum | Too restrictive; free-text `group` + conventions |
| Node-level `templates` | Reserved — preset config on add (distinct from `examples`) |
| Header `moduleVersion` | Reserved — plugin semver in catalog JSON (`"1.2.0"`); marketplace, diagnostics, compatibility; processor does not emit in v1 |

## Runtime API

`ExtensionCatalogLoader.loadMerged()` → typed **`ExtensionCatalog`** (not `JsonNode`). Immutable; editor-only metadata. See [ARCHITECTURE.md](ARCHITECTURE.md).

## Reserved: node templates (post-v1)

Not in v1. Documented direction for polished workflow builders:

```java
// Planned — not in @OloNode today
templates = {
    "Summarize Document",
    "Generate Release Notes",
    "Translate Text"
}
```

**UX goal:** when adding a Prompt node, the editor offers template choices and auto-populates the `prompt` configuration field.

Distinct from `examples` (discovery bullets) — templates would carry preset configuration values.

## Reserved: property validation (post-v1)

```java
// Planned
@OloProperty(name = "maxResults", type = OloPropertyType.NUMBER, min = 1, max = 100)
```

For fields like `temperature`, `retryCount`, `maxResults`.
